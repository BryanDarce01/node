# 1. Separar la API en resources (recursos)

La imagen dice:

- **“Separate API into logical resources”**

## ¿Qué es un resource?

Un resource (recurso) es simplemente:

- Un “tipo de dato” o entidad del sistema

Ejemplos de la imagen:

- tours
- users
- reviews

![pasos 1 y 2](../../img/rest/01.%20rest.png)

Si estamos haciendo una app, tenemos algo así:

- Usuarios 👤 → users
- Productos 🛒 → products
- Pedidos 📦 → orders

Cada uno es un recurso

## En Express se ve así:

En este ejemplo se separa la API por recursos.

``` javascript
app.get('/users', ...)
app.get('/tours', ...)
app.get('/reviews', ...)
```

❌ Error común

Hacer endpoints como:

```
/getUsersData
/doSomethingWithTours
```

- Eso NO es REST

# 2. Usar URLs basadas en recursos (no acciones)

La imagen dice:

- **“Expose structured, resource-based URLs”**

Idea clave:

- Las URLs deben representar cosas (sustantivos)

- NO acciones (verbos)

### MAL (lo que muestra la imagen)

Aquí se están metiendo acciones en la URL

```

/addNewTour
/getTour
/deleteTour

```

### BIEN (REST)

```

/tours
/tours/123
/users
/users/45

```

- Solo recursos (nombres)

# 3. Usar HTTP methods (verbos)

![metodos_http](../../img/rest/02.%20http_methods.png)

La regla es:

- NO usar verbos en la URL

- Usar verbos en el método HTTP

## ¿Por qué?

Porque en REST:

- La URL = el recurso (sustantivo)

- El método HTTP = la acción (verbo)

## 3.1 CRUD (lo que está a la derecha)

CRUD = operaciones básicas de datos:

| Operación | Método HTTP | Ejemplo    |
| --------- | ----------- | ---------- |
| Create    | POST        | `/tours`   |
| Read      | GET         | `/tours/7` |
| Update    | PUT / PATCH | `/tours/7` |
| Delete    | DELETE      | `/tours/7` |

## En Express sería así:

``` javascript

app.post('/tours', ...)       // crear
app.get('/tours/:id', ...)   // leer
app.patch('/tours/:id', ...) // actualizar
app.delete('/tours/:id', ...) // eliminar

```

## 3.2 Diferencia entre PUT vs PATCH

**1. PUT = reemplazar TODO**

- Se envía el objeto completo
- Se reemplaza el recurso entero

### Ejemplo:

Antes:

``` json

{
  "name": "Tour A",
  "price": 100
}

```
Request (PUT):

``` json

{
  "name": "Tour Nuevo",
  "price": 200
}

```

- Resultado: TODO cambia

- Si no se manda un campo, puede perderse

**2. PATCH = actualizar PARTE**

- Solo se envía lo que queremos cambiar

Ejemplo:

``` json
{
  "price": 150
}
```

Resultado:

``` json
{
  "name": "Tour A",
  "price": 150
}
```

- Solo cambió el precio

## 

- PUT = reemplazar

- PATCH = modificar

## Resumen punto 3

- No poner verbos en la URL

- Usar métodos HTTP para acciones

- CRUD = Create, Read, Update, Delete

- PUT = reemplaza todo

- PATCH = cambia una parte

# 4. Send data as JSON (usually)

![json](../../img/rest/03.%20json.png)

Significa:

- Las APIs REST normalmente envían y reciben datos en formato JSON

## ¿Qué es JSON?

**JSON = JavaScript Object Notation**

Es básicamente:
- una forma de representar datos como texto

Ejemplo:

``` json

{
  "id": 5,
  "tourName": "The Park Camper",
  "rating": "4.9"
}

```

## 4.1 Partes del JSON (lo que marca la imagen)


### 1. String (clave)

``` json
"id"
```

- Siempre va entre comillas

### 2. Value (valor)

``` json
5
"The Park Camper"
"4.9"
```

### 3. Key-value pair

``` json
"id": 5
```

- clave + valor

### 4. Object

``` json
{
  "id": 5
}

```
- Un conjunto de datos

### 5. Array

``` json
"guides": [
  {
    "name": "Steven Miller"
  }
]
```

- Lista de cosas

## 4.2 ¿Por qué JSON?

Porque es:

- Fácil de leer

- Fácil de enviar

- Compatible con JS (perfecto para Node)

### En Express

``` javascript

app.get('/tours/5', (req, res) => {
  res.json({
    id: 5,
    name: 'Tour A'
  })
})

```

- `res.json()` automáticamente lo convierte a JSON

### Recibir el JSON

``` javascript

app.post('/tours', (req, res) => {
  console.log(req.body)
})

```

## 4.3 Lo de “Response formatting” (lado derecho)

Nos dice que:

- No solo mandamos JSON

- También podemos darle estructura

## 4.4 ¿Qué es eso de JSend?

- Es una convención (regla opcional) para organizar respuestas

Siempre tiene:

- `status`

- `data`

Otras formas (la imagen lo menciona)

- JSON:API

- OData

Son estándares más complejos

Pero en la práctica la mayoría usa JSON simple o estilo JSend

## Resumen JSON

- JSON = formato para enviar datos

- Es texto estructurado (clave: valor)

- Express usa `res.json()`

- Podemos estructurar respuestas (ej: `status`, `data`)

# 5. Stateless

Significa **“Sin estado”**

- **El servidor NO guarda información entre requests**

![stateless](../../img/rest/04.%20stateless.png)

Traducción real

- **Cada request debe traer TODO lo necesario**

- El servidor:

  ❌ NO recuerda **requests** anteriores

  ❌ NO guarda “en qué estaba”

  ✅ Solo responde con lo que se manda AHORA

## Ejemplo de la imagen

### ❌ MAL (con estado en el servidor)

```
GET /tours/nextPage
```

### ¿Qué pasa aquí?

El servidor tiene que saber:

```
currentPage = 5
```

Luego calcula:

```
nextPage = 6
```

Problema:

- El servidor está recordando estado


### ✅ BIEN (stateless)

```
GET /tours/page/6
```

Aquí:

- El cliente manda todo

- El servidor no tiene que recordar nada

## Otro ejemplo

### ❌ MAL

```
GET /cart/addItem
```

- ¿Qué carrito?

- ¿De quién?

El servidor tendría que recordar sesión

### ✅ BIEN

```
POST /cart
Authorization: Bearer token123
```

Aquí el cliente manda:

- quién es (token)

- qué quiere hacer

## Nota

- El cliente guarda el estado

- El servidor solo responde

# Conexión con TODO lo anterior

Ya tenemos:

1. Recursos → `/tours`
2. URLs → `/tours/5`
3. Métodos → `GET`
4. JSON → datos

Ahora:

5. Stateless → cada request es independiente

# Resumen

- Stateless = el servidor no guarda estado

- Cada request trae toda la info

- No depende de requests anteriores

- Hace la API más escalable

## Analogía final

Es como una ventanilla de banco 🏦:

Cada vez que llegamos:

- Damos el ID

- Decimos qué queremos

El cajero no recuerda la visita anterior

Siempre damos toda la info
