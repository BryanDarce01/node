Cuando miramos el símbolo `^` en `package.json`, pensamos que `npm` siempre **instala la versión más nueva**, pero en realidad `package-lock.json` controla la versión exacta instalada.

# 1. Qué es package-lock.json

Cuando instalamos algo:

```
npm install
```

`npm` crea:

```
package.json
package-lock.json
```
`package.json`

- Define las reglas de versiones permitidas.

## Ejemplo

``` JSON
"dependencies": {
  "nodemon": "^3.0.0"
}
```

Esto significa:

- "puedes instalar cualquier versión 3.x.x".

## Pero

`package-lock.json`

Guarda la versión exacta que se instaló.

Ejemplo:

``` JSON
"nodemon": {
  "version": "3.0.2"
}
```

Entonces el proyecto queda congelado en esa versión exacta.

# 2. Qué problema soluciona

Imaginemos este escenario:

Hoy ejecuto:

```
npm install
```
y `npm` instala:

```
nodemon 3.0.2
```

Pero mañana sale:

```
nodemon 3.1.0
```

Si no existiera `package-lock.json`, otro desarrollador podría instalar:

```
nodemon 3.1.0
```

y el proyecto podría comportarse diferente.

# 3. Qué hace el `lock file`

Con `package-lock.json`:

Cuando alguien ejecuta:

```
npm install
```

`npm` ignora el rango del `^` y usa la versión exacta del `lock`.

Ejemplo:

```
package.json      → ^3.0.0
package-lock.json → 3.0.2
```
`npm` instala:

```
3.0.2
```

# 4. Cuándo cambia `package-lock.json`

Solo cambia cuando ejecutamos algo como:

```
npm update
```

o

```
npm install nodemon@latest
```

Entonces `npm` puede actualizar:

```
3.0.2 → 3.1.0
```

y actualiza el `lock file`.


# 5. Resumen

| Archivo             | Qué hace                     |
| ------------------- | ---------------------------- |
| `package.json`      | rango permitido de versiones |
| `package-lock.json` | versión exacta instalada     |
