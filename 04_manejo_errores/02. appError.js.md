# Esta clase se usa para crear errores

## ¿Qué es `AppError`?

`appError` es el nombre que le pongo al archivo que contiene la clase.

``` javascript

class AppError extends Error {
    constructor(message, statusCode){
        super(message)

        this.statusCode = statusCode;
        this.status = `${statusCode}`.startsWith('4') ? 'fail' : 'error';
        this.isOperational = true

        Error.captureStackTrace(this, this.constructor)
    }
}

module.exports = AppError;

```

Aquí estamos creando nuestro propio **tipo de error personalizado**

### En vez de usar:

``` javascript

new Error('algo salió mal')

```
Ahora usamos:

``` javascript

new AppError('algo salió mal', 404)

```

## ¿Por qué hacer esto?

Porque el `Error` normal:

- ❌ no tiene `statusCode`

- ❌ no diferencia tipos de error

- ❌ no sabe si es operacional o bug

`AppError` soluciona eso

# Desgloce de la clase `AppError`

## Paso 1.

``` javascript

class AppError extends Error

```

- En esta parte lo que estamos haciendo es crear un tipo de error más inteligente basado en el error nativo de JavaScript.

- `Error` ya existe en JS. Aquí lo estamos **extendiendo**.

## 2. El constructor

``` javascript

constructor(message, statusCode)

```
Cuando hacemos esto:

``` javascript

new AppError('No encontrado', 404)

```

- Se ejecuta el constructor.

## 3. ¿Por qué `super(message)`?

## Al crear la clase

``` javascript

class AppError extends Error

```

- No estamos creando un error todavía.

- Solo estamos definiendo una clase.

### Entonces… ¿qué hace `super(message)`?

- Inicializa la parte `Error` de la clase.

``` javascript

super(message)

```

En otras palabras, le decimos a JS:

***Ejecuta el constructor de `Error` y crea correctamente este error”***

- Esto es OBLIGATORIO cuando extendemos una clase.

- Hacemos esto porque `message` es un atributo de `Error` y esta es la forma de inicializarlo.

### ¿Por qué pasa eso?

Porque en JavaScript:

- Cuando usamos `extends`, el objeto **no está listo hasta que llamemos a** `super()`

## Sin `super` pasaría esto

``` javascript

class AppError extends Error {
  constructor(message) {
    this.message = message // ❌ ERROR
  }
}

```

### Con `super(message)`

``` javascript

super(message)

```

Hace esto internamente:

- crea el objeto base

- define `this.message`

- define `this.stack`

- Hace que `instanceof Error` funcione

- inicializa el error correctamente

- Hace que Express lo trate como error real

## 4. `statusCode`

``` javascript

this.statusCode = statusCode;

```
- Guardamos el código **HTTP**

Ejemplo:

``` javascript
404, 500, 401
```


## 5. El `startsWith('4')`

``` javascript

this.status = `${statusCode}`.startsWith('4') ? 'fail' : 'error';

```

### Paso 5.1: convertir a string

``` javascript

${statusCode}

```

- Porque `startsWith` solo funciona en strings

### Paso 5.2: `startsWith('4')`

``` javascript

"404".startsWith('4') → true
"500".startsWith('4') → false

```

### Paso 5.3: Operador ternario

``` javascript

condición ? valorSiTrue : valorSiFalse

```

### Resultado

| statusCode | status  |
| ---------- | ------- |
| 404        | 'fail'  |
| 401        | 'fail'  |
| 500        | 'error' |
| 503        | 'error' |

### ¿Por qué esto?

- Convención en APIs:

  - 4xx → error del cliente → "fail"
  - 5xx → error del servidor → "error"

## 6. `isOperational = true`

``` javascript

this.isOperational = true

```
Marca el error como:

- “Este error es esperado”

### Sirve para diferencia

| Tipo de error               | isOperational |
| --------------------------- | ------------- |
| Usuario no existe           | true          |
| Password incorrecto         | true          |
| Bug (`undefined.something`) | false         |


## 7. `Error.captureStackTrace`

``` javascript

Error.captureStackTrace(this, this.constructor)

```

### ¿Qué es el stack trace?

Cuando hay un error:

``` bash

Error: algo salió mal
    at getUser (userController.js:10)
    at ...

```

- Eso es el stack trace (la ruta del error)

### ¿Qué hace esta línea?

Le dice a Node:

- “Captura el error, pero NO incluyas esta clase (`AppError`) en el stack”

Resultado:

Sin esto:

```

at new AppError (...)
at getUser (...)

```
Con esto:

```

at getUser (...)

```

- Más limpio para debug

## Resumen

```

AppError = Error + statusCode + tipo de error + bandera de seguridad + stack limpio

```

# Todo junto

Cuando hacemos:

``` javascript

new AppError('No encontrado', 404)

```

Internamente pasa esto:

``` javascript

{
  message: 'No encontrado',
  statusCode: 404,
  status: 'fail',
  isOperational: true,
  stack: '...'
}

```

### Cuándo usar esta clase

Siempre que el error sea:

- esperado

- controlado

- parte del negocio


No usar para bugs:

``` javascript

undefined.something // ❌ no usar AppError

```

Frase clave

- "`AppError` convierte errores simples en errores inteligentes para APIs"

# El `errorGloblalHandler` se usa para responder al cliente el error que creamos con `appError.js`

Archivo que va en la carpeta controllers, ``errorGloblalHandler``

``` javascript

module.exports = (err, req, res, next) => {
  err.statusCode = err.statusCode || 500;
  err.status = err.status || 'error';

  res.status(err.statusCode).json({
    status: err.status,
    message: err.message
  });
};

```

## Uso en `app.js`

``` javascript

const globalErrorHandler = require('./controllers/errorController');

app.use(globalErrorHandler);

```

En vez de :

``` javascript

app.use((err, req, res, next) => { ... })

```

# ¿Cómo encaja con todo?

Flujo completo:

```
1. Usuario hace request
2. Router → Controller
3. Algo falla → next(new AppError(...))
4. app.all('*') (si no existe ruta)
5. globalErrorHandler (este archivo)
6. Response al cliente
```
