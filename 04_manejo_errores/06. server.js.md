# Manejo Global de Promesas y Errores Síncronos

``` javascript

const port = process.env.PORT || 3000;

const server = app.listen(port, () => {
  console.log(`App running on port ${port}...`);
});

// Manejo de promesas no controladas
process.on('unhandledRejection', err => {
  console.log(err.name, err.message);
  console.log('UNHANDLED REJECTION! 💥 Shutting down...');

  server.close(() => {
    process.exit(1);
  });
});

// Manejo de errores síncronos que no fueron capturados
process.on('uncaughtException', err => {
  console.log('UNCAUGHT EXCEPTION! 💥 Shutting down...');
  console.log(err.name, err.message);
  process.exit(1);
});

```

# A. `unhandledRejection`

# 1. Definir el puerto

``` javascript

const port = process.env.PORT || 3000;

```

Significa:

- **Usa el puerto del entorno, y si no existe, usa 3000**

## ¿Qué hace?

- `process.env.PORT`-  → variable de entorno (configuración externa)

- `|| 3000`  → valor por defecto

### Ejemplo

```

PORT=8000 node server.js

```

- entonces usa 8000

- si no → 3000

# 2. Levantar el servidor

``` javascript

const server = app.listen(port, () => {
  console.log(`App running on port ${port}...`);
});

```

## ¿Qué pasa aquí?

- `app.listen(...)` inicia el servidor HTTP

- queda “escuchando” requests

## Muy importante

``` javascript

const server = app.listen(...)

```

- `server` NO es Express

- es el **servidor HTTP de Node**

Esto te permite luego hacer:

``` javascript

server.close()

```

# 3. `process.on('unhandledRejection')`

``` javascript

process.on('unhandledRejection', err => {

```
  
## ¿Qué es `process`?

- Es el objeto global de Node.js

- Representa el proceso en ejecución

## ¿Qué evento es este?

```

unhandledRejection

```

Significa:

- **Una promesa falló y nadie hizo `.catch()`**

## Ejemplo

``` javascript

Promise.reject(new Error('Boom'));

```

o:

``` javascript

async function test() {
  throw new Error('Error');
}

test(); // ❌ sin catch

```

## ¿Por qué es peligroso?

Porque:

- el error NO pasa por Express

- la app puede quedar en estado inconsistente

# 4. El handler

``` javascript

console.log(err.name, err.message);

```

Imprime algo como:

```

Error Something went wrong

```

``` javascript

console.log('UNHANDLED REJECTION! 💥 Shutting down...');

```

- Es solo un log informativo

# 5. Parte más importante

``` javascript

server.close(() => {
  process.exit(1);
});

```

## ¿Qué hace `server.close()`?

Deja de aceptar nuevas requests

Pero:

- deja terminar las que ya están en proceso

### Luego:

``` javascript

process.exit(1);

```

- Mata el proceso de Node

## ¿Por qué 1?

```

0 = éxito
1 = error

```

## ¿Por qué cerrar la app?

Esto es clave

Cuando hay un error no manejado:

- no sabemos en qué estado quedó la app

- puede haber:

  - conexiones abiertas

  - datos corruptos

  - lógica rota

## Importante: diferencia con Express

| Tipo de error          | Quién lo maneja    |
| ---------------------- | ------------------ |
| `next(err)`            | Express middleware |
| `throw` en async route | catchAsync         |
| Promesa sin catch      | process.on(...)    |

## Resumen

- `app.listen` → levanta el servidor

- `server` → te permite cerrarlo

- `unhandledRejection` → captura promesas sin catch

- `server.close()` → cierra correctamente

- `process.exit(1)` → termina el proceso

# B. `uncaughtException`

## Manejo de errores síncronos que no fueron capturados.

``` javascript

process.on('uncaughtException', err => {
  console.log('UNCAUGHT EXCEPTION! 💥 Shutting down...');
  console.log(err.name, err.message);
  process.exit(1);
});

```

## 1. ¿Qué es `uncaughtException`?

Es un evento de Node que significa:

- **Ocurrió un error y nadie lo atrapó con `try/catch`**

## Ejemplo

``` javascript

console.log(x); // ❌ x no existe

```

Esto lanza:

```

ReferenceError: x is not defined

```

Y como no hay `try/catch` → se vuelve `uncaughtException`

## Diferencias

| Tipo de error      | Ejemplo                |
| ------------------ | ---------------------- |
| `uncaughtException`  | error síncrono (throw) |
| `unhandledRejection` | promesa sin `.catch()` |

## 2. `process.on(...)`

``` javascript

process.on('uncaughtException', err => {

```
Esto le dice a Node:

- **Si pasa esto, ejecuta esta función**

## 3. Logs

``` javascript

console.log('UNCAUGHT EXCEPTION! 💥 Shutting down...');
console.log(err.name, err.message);

```

Ejemplo:

```

UNCAUGHT EXCEPTION! 💥 Shutting down...
ReferenceError x is not defined

```

## 4. Parte crítica

``` javascript

process.exit(1);

```

- Mata el proceso inmediatamente

## ¿Por qué NO usamos `server.close()` aquí?

A diferencia de `unhandledRejection`:

**Aquí el error es síncrono y grave**

- el proceso ya está en estado inconsistente

- puede que Express ni siquiera haya arrancado bien

💀 No es seguro seguir ejecutando nada

## Diferencia entre ambos


### 1. `uncaughtException`

Error síncrono (inmediato)

- → puede romper todo el proceso

- → salir inmediatamente


### 2. `unhandledRejection`

Error async (promesas)

- → puedes cerrar el server de forma controlada

- → luego salir
