# Stack Trace
Es básicamente el **historial de llamadas de funciones** que se ejecutaron antes de que ocurriera un error.

En otras palabras:

- Es como un “rastro” que nos dice por dónde pasó el código hasta que algo falló.

## ¿Qué contiene un stack trace?

Cuando ocurre un error, JavaScript guarda algo como esto:

```

Error: No encontrado
    at getUser (controllers/userController.js:10:15)
    at processRequest (routes/userRoutes.js:5:3)
    at app.js:20:1

```

Esto significa:

- ❌ Ocurrió un error: `"No encontrado"`

- 📍 En la función `getUser`

- 📍 Que fue llamada por `processRequest`

- 📍 Que fue llamada desde `app.js`

Es literalmente una **pila (stack)** de llamadas, de la más reciente hacia atrás.

## ¿Por qué se llama “stack”?

Porque sigue la estructura de datos **stack (pila)**:

- La última función que se ejecuta → es la primera que aparece.

- Va bajando hacia atrás en el flujo del programa.

## En JavaScript (Node.js)

Todos los errores (`Error`) tienen una propiedad:

``` javascript

try {
  throw new Error('Algo salió mal')
} catch (err) {
  console.log(err.stack)
}

```

- `err.stack` = el stack trace completo

# En el caso  [`appError`](16.%20appError.js.md)

``` javascript

class AppError extends Error {
  constructor(message, statusCode) {
    super(message)
    this.statusCode = statusCode
  }
}

```

El `super(message)` es clave porque:

- Inicializa el `Error`

- Y automáticamente crea el **stack trace**

Si no llamamos a `super`:

- ❌ No tendríamos correctamente el `message` ni el `stack`

## ¿Para qué sirve en la vida real?

Principalmente para **debugging**:

- Saber **dónde ocurrió el error**

- Ver **qué funciones lo causaron**

- Seguir el flujo del programa

# Ahora en

``` javascript

Error.captureStackTrace(this, this.constructor)

```

## ¿Qué hace Error.`captureStackTrace`?

Es un método propio del motor de Node.js (V8) que:

- Crea manualmente el `stack trace` en un objeto (en este caso `this`).

Es como decirle:

- “Oye, genera el stack trace para este error personalizado”

## ¿Qué significan los parámetros?

``` javascript

Error.captureStackTrace(target, constructorOpt)

```

### 1. `target` → `this`

Es el objeto donde queremos guardar el stack

- En este caso: la instancia de `AppError`

### 2. `constructorOpt` → `this.constructor`

Esto es lo importante:

Le dice a Node:

- “Empieza el stack trace **después de este constructor**”

## ¿Por qué se pasa `this.constructor`?

Sin eso, el **stack** incluiría algo así:

```

at new AppError (...)
at ...

```

Pero eso no es útil para nosotros, porque:

- No necesitamos ver el constructor del error.

- Queremos ver **dónde se lanzó el error en el código**.

## Ejemplo

### ❌ Sin `captureStackTrace`

``` javascript

class AppError extends Error {
  constructor(message) {
    super(message)
  }
}

```

Stack:

```

Error: Algo salió mal
    at new AppError (appError.js:2:3) ❌ ruido innecesario
    at getUser (controller.js:10:5)

```

### ✅ Con `captureStackTrace`

``` javascript

class AppError extends Error {
  constructor(message) {
    super(message)
    Error.captureStackTrace(this, this.constructor)
  }
}

```

Stack:

```

Error: Algo salió mal
    at getUser (controller.js:10:5) ✅ limpio

```

## Entonces, en resumen:

``` javascript

Error.captureStackTrace(this, this.constructor)

```

Hace esto:


- ✅ Genera el stack trace para tu error personalizado.

- ✂️ Elimina del stack la parte del constructor (`AppError`)

- 🧼 Deja un stack más limpio y útil para debugging
