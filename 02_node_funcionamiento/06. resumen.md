# 4 Conceptos Clave

```

Non-blocking I/O
       ↓
Event Driven Architecture
       ↓
Callbacks
       ↓
Event Loop

```

# 1. Non-blocking I/O (la base de todo)

Primero: **I/O significa Input / Output**, por ejemplo:

- leer archivos

- hacer requests HTTP

- consultar una base de datos

- leer red

- escribir en disco

En muchos lenguajes esto es **blocking**.

**Ejemplo blocking**
```

const data = readFile('file.txt');
console.log(data);

```

Aquí pasa esto:

```

leer archivo
↓
esperar...
↓
esperar...
↓
termina
↓
seguir ejecutando código

```

El programa se **queda detenido esperando.**

**Node hace lo contrario: Non-blocking**

En Node:

``` javascript
fs.readFile('file.txt', (err, data) => {
  console.log(data);
});

```

Aquí pasa esto:

```

inicia lectura del archivo
↓
Node NO espera
↓
sigue ejecutando otras cosas
↓
cuando termina → dispara evento

```

Esto permite que un **solo hilo maneje miles de operaciones**.

# 2. Event Driven Architecture

Como **Node** no espera, necesita otra forma de reaccionar cuando algo termina.

Por eso usa **eventos**.

Flujo:

```

operación inicia
↓
operación termina
↓
se emite un evento
↓
alguien escucha el evento
↓
se ejecuta código

```

Ejemplo:

``` javascript

server.on('request', (req, res) => {
  res.end('Hola');
});

```

Aquí:

```

evento → request
listener → server.on()
acción → callback

```

**Node** está básicamente diciendo:

```

cuando pase esto → ejecuta esto

```

# 3. Callbacks

Las **callbacks** son las funciones que se ejecutan cuando ocurre el evento.

Ejemplo:

``` javascript
fs.readFile('file.txt', (err, data) => {
  console.log(data);
});

```

La **callback** es:

``` javascript

(err, data) => {
  console.log(data);
}

```

Esta función **no se ejecuta inmediatamente**.

Solo se ejecuta **cuando termina la operación.**

Flujo:

```

leer archivo
↓
termina
↓
ejecutar callback

```

# 4. Event Loop (el cerebro de Node)

Aquí entra el **Event Loop**.

El **Event Loop** es el sistema que:

```

revisa eventos terminados
↓
toma su callback
↓
la ejecuta

```

Es como un **administrador de tareas**.

Flujo simplificado:

```

Event Queue (callbacks esperando)
        ↓
Event Loop revisa
        ↓
si el stack está libre
        ↓
ejecuta callback

```

# 5. Cómo se conectan los 4

Ahora juntamos todo.

## Paso 1 — Inicia una operación I/O

``` javascript

fs.readFile('file.txt', callback);

```

Esto es **Non-blocking I/O**.

**Node** dice:

```

empieza a leer archivo
yo sigo trabajando

```

## Paso 2 — Termina la operación

Cuando el sistema termina de leer el archivo:

```

se emite un evento

```

Esto es **Event Driven Architecture**.

## Paso 3 — Se registra la callback

La función que pasaste:

``` javascript

(err, data) => {...}

```

Es una **callback**.

## Paso 4 — Event Loop ejecuta la callback

El **Event Loop hace**:

```

operación terminada
↓
callback a la cola
↓
stack libre
↓
ejecutar callback

```

## 6. Ejemplo completo en flujo

Código:

``` javascript

console.log('Inicio');

fs.readFile('file.txt', (err, data) => {
  console.log('Archivo leído');
});

console.log('Fin');

```

Orden real:

```

Inicio
Fin
Archivo leído

```

Flujo interno:

```

Inicio
↓
inicia readFile
↓
Node no espera
↓
Fin
↓
archivo termina de leerse
↓
evento
↓
callback a la cola
↓
Event Loop ejecuta callback
↓
Archivo leído

```

# 7. La arquitectura completa de Node

Todo junto:

```

Application code
        ↓
Non-blocking I/O
        ↓
Event Driven Architecture
        ↓
Callbacks
        ↓
Event Loop
        ↓
Execution

```

Por eso **Node** puede manejar:

```

miles de requests
con un solo hilo

```

# 8. Regla mental

**Node** piensa así:

```

No esperes operaciones
↓
registra qué hacer cuando terminen
↓
Event Loop ejecutará ese código

```
