# Optimización de consultas en MongoDB: Los índices (indexes).

## Cuando hacemos `userSchema.index(...)`

``` javascript

userSchema.index({ email: 1 });

```

le decimos a MongoDB:

- "Crea un índice para el campo email."

## 1. Sin índice

Si tenemos una colección con 1 millón de usuarios:

``` javascript

[
  { email: 'a@gmail.com' },
  { email: 'b@gmail.com' },
  { email: 'c@gmail.com' },
  ...
]

```

Y hacemos:

``` javascript

User.findOne({ email: 'b@gmail.com' });

```

MongoDB tendría que revisar:

```

Usuario 1
Usuario 2
Usuario 3
Usuario 4
...

```

hasta encontrar el correcto.

A esto se le llama:

```

COLLSCAN (Collection Scan)

```

- Escanear toda la colección.

## 2. Con índice

Si hacemos:

``` javascript

userSchema.index({ email: 1 });

```

MongoDB crea una estructura parecida a:

```

a@gmail.com -> documento A
b@gmail.com -> documento B
c@gmail.com -> documento C

```

Entonces cuando buscamos:

``` javascript

User.findOne({ email: 'b@gmail.com' });

```

- MongoDB va directamente al registro.

- Mucho más rápido.

### ¿Qué significa el `1`?

``` javascript

userSchema.index({ email: 1 });

```

El `1` significa:

- Orden ascendente

### También existe:

``` javascript

userSchema.index({ email: -1 });

```

que significa:

- Orden descendente

## 3. Índices compuestos

También podemos hacer:

``` javascript

userSchema.index({
  role: 1,
  createdAt: -1
});

```

MongoDB indexará ambos campos.

Útil para consultas como:

``` javascript

User.find({ role: 'admin' })
    .sort({ createdAt: -1 });

```

## 4. ¿Cuándo crear índices?

Campos que vayamos a consultar frecuentemente:

``` javascript

email
slug
username
role
createdAt

```

# ¿Qué hace `.explain()`?

`.explain()` permite ver cómo MongoDB ejecuta una consulta.

Ejemplo:

``` javascript

User.find({ email: 'john@gmail.com' }).explain();

```

MongoDB devuelve información de ejecución.

### Sin índice

Sale algo parecido a:

``` javascript

{
  stage: "COLLSCAN"
}

```

Significa:

```

Collection Scan

```

- MongoDB revisó toda la colección.

- Mala señal para colecciones grandes.

### Con índice

Después de:

``` javascript

userSchema.index({ email: 1 });

```

podríamos ver:

``` javascript

{
  stage: "IXSCAN"
}

```

Significa:

```

Index Scan

```

- MongoDB usó el índice.

- Buena señal.

## Los campos más importantes de `explain()`

### executionStats

Muchas veces se usa:

``` javascript

User.find({ email: 'john@gmail.com' })
    .explain('executionStats');

```

### totalDocsExamined

``` javascript

totalDocsExamined: 1000000

```

- MongoDB revisó 1 millón de documentos.

- Muy malo.

### totalKeysExamined

``` javascript

totalKeysExamined: 1

```

- MongoDB revisó solo una entrada del índice.

- Muy bueno.

## Importante en Mongoose

Si definimos:

``` javascript

email: {
  type: String,
  unique: true
}

```

Mongoose crea automáticamente un índice único:

``` javascript

{
  email: 1
}

```

Por eso muchas veces no vamos a ver:

``` javascript

userSchema.index({ email: 1 });

```

porque el índice ya existe gracias al:

``` javascript

unique: true

```

# Resumen

- `userSchema.index({ email: 1 })` crea un índice para acelerar búsquedas.

- `1` = orden ascendente, `-1` = descendente.

- Sin índice MongoDB hace `COLLSCAN`(escanea toda la colección).

- Con índice MongoDB hace `IXSCAN` (usa el índice).

- `.explain()` muestra cómo MongoDB ejecutó la consulta.

- Lo más importante de `explain()` suele ser:

    - `stage` (`COLLSCAN` o `IXSCAN`)

    - `totalDocsExamined`

    - `executionTimeMillis`

Muchos problemas de rendimiento se solucionan simplemente creando los índices correctos.

# Vamos con el `this.constructor`

[reviewsModel.js](https://github.com/jonasschmedtmann/complete-node-bootcamp/blob/master/4-natours/after-section-12/models/reviewModel.js)

``` javascript
// review / rating / createdAt / ref to tour / ref to user
const mongoose = require('mongoose');
const Tour = require('./tourModel');

const reviewSchema = new mongoose.Schema(
  {
    review: {
      type: String,
      required: [true, 'Review can not be empty!']
    },
    rating: {
      type: Number,
      min: 1,
      max: 5
    },
    createdAt: {
      type: Date,
      default: Date.now
    },
    tour: {
      type: mongoose.Schema.ObjectId,
      ref: 'Tour',
      required: [true, 'Review must belong to a tour.']
    },
    user: {
      type: mongoose.Schema.ObjectId,
      ref: 'User',
      required: [true, 'Review must belong to a user']
    }
  },
  {
    toJSON: { virtuals: true },
    toObject: { virtuals: true }
  }
);

reviewSchema.index({ tour: 1, user: 1 }, { unique: true });

reviewSchema.pre(/^find/, function(next) {
  // this.populate({
  //   path: 'tour',
  //   select: 'name'
  // }).populate({
  //   path: 'user',
  //   select: 'name photo'
  // });

  this.populate({
    path: 'user',
    select: 'name photo'
  });
  next();
});

reviewSchema.statics.calcAverageRatings = async function(tourId) {
  const stats = await this.aggregate([
    {
      $match: { tour: tourId }
    },
    {
      $group: {
        _id: '$tour',
        nRating: { $sum: 1 },
        avgRating: { $avg: '$rating' }
      }
    }
  ]);
  // console.log(stats);

  if (stats.length > 0) {
    await Tour.findByIdAndUpdate(tourId, {
      ratingsQuantity: stats[0].nRating,
      ratingsAverage: stats[0].avgRating
    });
  } else {
    await Tour.findByIdAndUpdate(tourId, {
      ratingsQuantity: 0,
      ratingsAverage: 4.5
    });
  }
};

reviewSchema.post('save', function() {
  // this points to current review
  this.constructor.calcAverageRatings(this.tour);
});

// findByIdAndUpdate
// findByIdAndDelete
reviewSchema.pre(/^findOneAnd/, async function(next) {
  this.r = await this.findOne();
  // console.log(this.r);
  next();
});

reviewSchema.post(/^findOneAnd/, async function() {
  // await this.findOne(); does NOT work here, query has already executed
  await this.r.constructor.calcAverageRatings(this.r.tour);
});

const Review = mongoose.model('Review', reviewSchema);

module.exports = Review;

```

## Puntos importantes

1. Métodos estáticos (`statics`)

- Document middleware (`save`)

- Query middleware (`findOneAndUpdate`, `findOneAndDelete`)

- `this.constructor`

## 1. ¿Qué hace `calcAverageRatings`?

Este método:

``` javascript

reviewSchema.statics.calcAverageRatings = async function(tourId) {

```

- Es un método **estático** del modelo Review.

Es equivalente a hacer:

``` javascript

Review.calcAverageRatings(tourId);

```

Su trabajo es:

- Buscar todas las reviews de un tour.

- Calcular:

  - Cantidad de reviews.

  - Promedio de ratings.

- Actualizar el documento Tour.

### Ejemplo

Hacemos de cuenta que el Tour tiene estas reviews:

``` javascript

[
  { rating: 5 },
  { rating: 4 },
  { rating: 3 }
]

```

La agregación devuelve:

``` javascript

[
  {
    _id: tourId,
    nRating: 3,
    avgRating: 4
  }
]

```

Entonces se actualiza:

``` javascript

Tour.findByIdAndUpdate(tourId, {
  ratingsQuantity: 3,
  ratingsAverage: 4
});

```

## 2. ¿Por qué guardar el promedio en Tour?

Porque sería muy costoso calcularlo cada vez.

En caso de que cada vez que alguien pide un tour, hacemos:

``` javascript

Review.aggregate(...)

```

para calcular el promedio.

Si tenemos en la base de datos:

```

1000 tours
100000 reviews

```

- sería lentísimo.

Por eso se usa el patrón:

```

Denormalización

```

- Guardar el promedio ya calculado dentro del Tour.

## 3. ¿Qué pasa cuando se crea una review?

Aquí:

``` javascript

reviewSchema.post('save', function() {
  this.constructor.calcAverageRatings(this.tour);
});

```

### ¿Qué es `this`?

Dentro de:

``` javascript

post('save')

```

`this` es el documento Review recién guardado.

### Ejemplo

``` javascript

{
  _id: '123',
  review: 'Excelente',
  rating: 5,
  tour: '999'
}

```

### ¿Qué es `this.constructor`?

Esto es JavaScript puro:

``` javascript

const arr = [];

arr.constructor

```

Resultado:

``` javascript

Array

```

### Otro ejemplo:

``` javascript

const fecha = new Date();

fecha.constructor

```

Resultado:

``` javascript

Date

```

### Con Mongoose:

``` javascript

this

```

es una instancia de:

``` javascript

Review

```

Por lo tanto:

``` javascript

this.constructor

```

es:

``` javascript

Review

```

Es decir:

``` javascript

this.constructor.calcAverageRatings(this.tour);

```

es prácticamente:

``` javascript

Review.calcAverageRatings(this.tour);

```

## 4. ¿Por qué no escribir directamente Review?

Porque en este punto:

``` javascript

reviewSchema.post('save', ...)

```

todavía no existe:

``` javascript

const Review = mongoose.model(...)

```

- El schema se está definiendo antes.

Por eso usan:

``` javascript

this.constructor

```

- para obtener el modelo automáticamente.

## 5. Ahora viene la parte complicada

### ¿Por qué hacen esto?

``` javascript

reviewSchema.pre(/^findOneAnd/, async function(next) {
  this.r = await this.findOne();
  next();
});

```

y luego:

``` javascript

reviewSchema.post(/^findOneAnd/, async function() {
  await this.r.constructor.calcAverageRatings(this.r.tour);
});

```

### El problema

Cuando hacemos:

``` javascript

Review.findByIdAndUpdate(...)

```
o

``` javascript

Review.findByIdAndDelete(...)

```

No tenemos acceso al documento en el middleware post.

### En `save` sí

``` javascript

reviewSchema.post('save', function() {
  console.log(this);
});

```

Aquí `this` es el documento.

### En `findOneAndUpdate` no

``` javascript

reviewSchema.post('findOneAndUpdate', function() {
  console.log(this);
});

```

Aquí `this` es el query.

No el documento.

### Por eso esto no funciona:

``` javascript

this.tour

```

porque `this` es algo parecido a:

``` javascript

Query {
   conditions: {...}
}

```

### La solución

Antes de actualizar:

``` javascript

reviewSchema.pre(/^findOneAnd/, async function(next) {

```

hacen:

``` javascript

this.r = await this.findOne();

```

### Supongamos que existe:

``` javascript

{
  _id: 123,
  rating: 5,
  tour: 999
}

```

Lo guardan en:

``` javascript

this.r

```

Después ocurre:

``` javascript

findOneAndUpdate

```

o

``` javascript

findOneAndDelete

```

Y finalmente:

``` javascript

post(/^findOneAnd/)

```

puede acceder a:

``` javascript

this.r

```

porque la guardaron antes.

### Entonces esta línea

``` javascript

await this.r.constructor.calcAverageRatings(this.r.tour);

```

significa:

- `this.r` → review encontrada antes del update/delete.

- `this.r.constructor` → modelo Review.

- `calcAverageRatings` → método estático del modelo.

- `this.r.tour` → tour al que pertenece la review.

Equivale prácticamente a:

``` javascript

await Review.calcAverageRatings(this.r.tour);

```

## 6. ¿Por qué hay que recalcular el promedio varias veces?

Porque el promedio puede cambiar cuando:

### 6.1. Se crea una review

``` javascript

save

```

```

4.5 → 4.6

```

### 6.2. Se actualiza una review

``` javascript

findByIdAndUpdate

```

```

4.6 → 4.2

```

### 6.3. Se elimina una review

``` javascript

findByIdAndDelete

```

```

4.2 → 4.0

```

Por eso hay tres escenarios:

``` javascript

post('save')

```

↓

``` javascript

pre(/^findOneAnd/)

```

↓

``` javascript

post(/^findOneAnd/)

```

Cada vez que una review cambia, el Tour vuelve a calcular:

``` javascript

ratingsAverage
ratingsQuantity

```

- para que siempre estén sincronizados con las reviews reales.
